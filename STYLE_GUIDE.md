# SpecQL Documentation Style Guide

## The SpecQL Documentation Personality

**Imagine**: A senior engineer who's also a great teacher. Smart but not showing off. Helpful but not condescending. Occasionally funny but never unprofessional.

**Voice Principles**:
1. **Clarity over cleverness**: Be witty, but never at the expense of understanding
2. **Concrete over abstract**: Show code examples, not just concepts
3. **Honest**: If something's experimental, say so
4. **Humor sparingly**: One pun per doc maximum üòÑ

## Structure

### Every Doc Starts with: What, Why, How

**What**: What is this feature/concept?
**Why**: Why would you use it? What's the benefit?
**How**: How do you implement it?

### Callouts for Important Information

```markdown
> **Note**: This is a note

> **Warning**: This is a warning

> **Tip**: This is a tip

> **Example**: This shows an example
```

### Code Examples

- **Always runnable**: Every code example should work when copy-pasted
- **Language specified**: Use proper code blocks with language
- **Commented**: Explain what each part does
- **Progressive**: Start simple, add complexity

## Formatting

### Headers
- **Sentence case**: "Getting started" not "Getting Started"
- **Consistent hierarchy**: H1 ‚Üí H2 ‚Üí H3, no skipping levels
- **Descriptive**: Headers should tell you what's in that section

### Lists
- **Parallel structure**: All items should have the same grammatical form
- **Consistent punctuation**: Either all end with periods, or none do
- **Action-oriented**: Start with verbs when describing steps

### Links
- **Relative paths**: Use relative links within the docs
- **Descriptive text**: "See the [getting started guide](01_getting-started/)" not "click here"
- **Future-proof**: Link to sections that are likely to persist

## Technical Writing Best Practices

### Be Specific
- "20 lines of YAML" not "a little YAML"
- "30 minutes" not "quickly"
- "97% less code" not "way less code"

### Use Active Voice
- ‚úÖ "SpecQL generates the code" not ‚ùå "The code is generated by SpecQL"
- ‚úÖ "Define your entity" not ‚ùå "The entity should be defined"

### Avoid Jargon
- Use "database table" not just "table"
- Explain acronyms on first use: "GraphQL (a query language for APIs)"
- Link to concepts: "See [Trinity Pattern](03_core-concepts/trinity-pattern.md) for details"

### Write for Scanning
- **Bold** key terms: "The **Trinity Pattern** uses three identifiers"
- *Italicize* emphasis: "This approach *scales better*"
- `Code` for technical terms: "Use the `email` rich type"
- Short paragraphs (3-5 sentences max)
- Bullet points for lists
